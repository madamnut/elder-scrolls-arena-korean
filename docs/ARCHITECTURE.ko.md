# 아키텍처와 런타임 구조

## 목적

원본 DOS 실행 파일의 게임 로직을 유지하면서 ASCII와 완성형 한글을 함께 표시한다. UTF-8을 DOS 엔진에 직접 넣지 않고, 빌드 시점에 프로젝트 전용 인코딩으로 변환한다.

## 구성 요소

```text
UTF-8 번역 원본
  │
  ├─ akc_codec.py ────────────────┐
  │                               │
  ├─ patch_acd.py → ACDKR.EXE     │
  ├─ build_questions.py           │
  │       └──────→ QUEST_KR.TXT   │
  └─ 기타 빌더 → BSA/FLC/DAT     │
                                  ▼
ACDKR.EXE ── INT 60h ── ARENAKR.COM ── EMS의 HANGUL.FNT/HANGUL12.FNT/HANGUL16.FNT
```

### `ACDKR.EXE`

- Deark로 해제한 Arena 1.07 CD 실행 파일에서 생성한다.
- 원본 폭 계산 함수와 그리기 함수 본체를 작은 `INT 60h` 스텁으로 교체한다.
- 한국어 전용 BSA, 인트로, 템플릿, 질문 파일 이름을 참조한다.
- 항상 검증된 언팩 원본에서 재빌드하며 설치된 결과물을 다시 입력으로 사용하지 않는다.

### `ARENAKR.COM`

- 게임 실행 전에 적재되는 DOS TSR이다.
- `DX=0`에서 문자열의 픽셀 폭을 계산한다.
- `DX=1`에서 문자열을 화면에 그린다.
- `DX=2`에서 현재 줄의 한글 포함 여부를 확인하고 행 이동값을 계산한다.
- ASCII는 Arena 글꼴의 높이·폭 표와 비트맵을 그대로 사용한다.
- AKC 한글은 현재 영문 글꼴 높이에 따라 `HANGUL.FNT`, `HANGUL12.FNT`, `HANGUL16.FNT`에서 읽는다. 각각의 원본 픽셀을 축소 없이 그린다.
- AKC 한글이 있는 줄은 최소 11픽셀, ASCII만 있는 줄은 원본 `font[0] + 1`만큼 이동한다.
- 최종 규칙은 한글화된 텍스트 블록 전체에 선택된 한글 뱅크의 행 이동값을 적용하는 것이다. 현재 줄 단위 AKC 감지는 전환 단계의 구현이며, 블록 안의 ASCII 전용 줄까지 같은 행간을 유지하도록 확장한다.
- HANGUL9 혼합 줄의 영문 글리프는 높이 차이만큼 아래로 이동해 9픽셀 한글 셀의 아래쪽에 맞춘다. 질문 빌더는 판정 태그가 혼자 ASCII 전용 줄로 분리되지 않게 검증한다.
- 직접 그리기 API는 원본과 같이 반환 시 `SI`를 소비한 NUL 다음으로 전진시킨다. 질문 스크롤은 이 부작용으로 NUL 분리된 다음 줄을 순회한다.
- 질문 스크롤의 전용 행 슬롯·부드러운 스크롤 위상·버퍼 이동량·새 줄 좌표를 HANGUL9 기준 11픽셀로 함께 패치한다. 48픽셀 표시 영역은 유지한다.
- 높이 9 이하의 영문 글꼴은 HANGUL9, 10~12는 HANGUL12, 13 이상은 HANGUL16에 대응한다.
- 게임이 사용하던 EMS 매핑을 저장하고 한글 출력 후 복구한다.

### 한글 글리프 뱅크

- 완성형 한글 11,172개를 담는다.
- 각 글리프는 32바이트 레코드다.
- 16KiB EMS 페이지 하나에 512글리프가 들어간다.
- `HANGUL.FNT`는 Galmuri9, `HANGUL12.FNT`는 물마루 Mono, `HANGUL16.FNT`는 Neo둥근모에서 생성한다.
- 세 파일은 각각 22 EMS 페이지이며 TSR은 총 66페이지를 한 번에 할당한다.
- 파일 하나는 완전한 16KiB 페이지 21개와 13,440바이트짜리 마지막 페이지로 구성된다. 로더는 마지막 페이지를 0으로 비운 뒤 사용 구간만 읽고 이 정확한 길이를 검증한다.
- 목표 구조는 9·12·16픽셀 뱅크를 분리하며, 9픽셀보다 작은 한글 뱅크는 만들지 않는다.
- 작은 영문 폰트 화면은 한글을 축소하는 대신 UI 영역과 행간을 조정한다.
- 세부 대응표와 배포 의무는 [한글 폰트 정책과 라이선스](FONTS.ko.md)에 기록한다.

## AKC 인코딩

ASCII 제어 문자 `TAB`, `LF`, `CR`과 `0x20–0x7F`는 1바이트로 보존한다. 완성형 한글은 2바이트다.

```text
lead  = 0x80 + (syllable_index >> 7)
trail = 0x80 + (syllable_index & 0x7F)
```

지원하지 않는 문자는 빌드 오류로 처리한다. 한자, 조합되지 않은 자모, 스마트 따옴표, 전각 문장부호 등을 번역 원본에 넣지 않는다.

## 원본 영어 글꼴

Arena 글꼴은 높이가 고정되고 문자별 전진 폭이 다른 가변폭 비트맵이다.

| 파일 | 높이 | 저장된 폭 범위 | 대표 용도 |
|---|---:|---:|---|
| `FONT_C.DAT` | 14 | 3–15 | 큰 제목 |
| `FONT_A.DAT` | 11 | 3–15 | 메뉴·캐릭터 생성 |
| `ARENAFNT.DAT` | 9 | 2–9 | 일반 인게임 글자 |
| `CHARFNT.DAT` | 8 | 2–5 | 캐릭터 정보 |
| `TEENYFNT.DAT` | 8 | 2–8 | 보조 UI |
| `FONT_D.DAT` | 7 | 2–6 | 작은 설명문 |
| `FONT_B.DAT` | 6 | 2–6 | 직업 질문 본문 |
| `FONT_S.DAT` | 5 | 2–6 | 지도·극소형 UI |

원본 줄 이동값은 대체로 `font[0] + 1`이다. 영문은 실제 획이 글리프 상자를 모두 채우지 않지만, 축소한 한글은 상자를 조밀하게 채우므로 같은 높이에서도 붙어 보일 수 있다.

## 텍스트 출처 분류

| 유형 | 예 | 처리 방식 |
|---|---|---|
| EXE 고정 문자열 | 캐릭터 생성 문구 | 원문 공간 안에서 AKC로 교체 |
| 느슨한 텍스트 파일 | `QUESTION.TXT` | 별도 한글 파일 생성 후 파일명 참조 변경 |
| BSA 내부 텍스트 | `*.INF`, `*.DAT` | 복호화·번역·재암호화 후 별도 BSA |
| 런타임 자막 | 컷신 대사 | 데이터 번역 + 한글 렌더러 |
| 이미지에 박힌 글자 | 타이틀·줄거리 슬라이드 | 이미지 편집 후 Arena 팔레트로 양자화 |

## 원본 보존 전략

- `GLOBAL.BSA` 대신 `GLOBAL_K.BSA`를 만든다.
- `ACD.EXE` 대신 `ACDKR.EXE`를 만든다.
- `INTRO.FLC` 대신 `INTKR.FLC`를 만든다.
- `TEMPLATE.DAT` 대신 `TEMPL_KR.DAT`를 만든다.
- `TEMPL_KR.DAT`의 각 해시 레코드는 `&` 뒤 패딩으로 원본 바이트 크기와 다음 헤더 오프셋을 보존한다.
- `QUESTION.TXT` 대신 `QUEST_KR.TXT`를 만든다.
- 원본 런타임 `ARENA`는 어떤 한글 파일로도 덮어쓰지 않는다.
- 설치 시 사용자 PC에서 `ARENA`를 복제한 한글 전용 런타임 `ARENA_KR`을 만든다.
- `TITLE.IMG`, `SCROLL03.IMG`, `TAMRIEL.MNU`처럼 원본과 같은 이름으로만 안정적으로 읽히는 파일은 `ARENA_KR` 안에서만 교체한다.
- 원본 실행 설정은 `ARENA`, 한글 실행 설정은 `ARENA_KR`을 각각 C:와 D:로 마운트한다.

이 분리는 느슨한 파일의 우선 로딩 때문에 필요하다. `GLOBAL_K.BSA`만 별도로 만들어도 원본 이름의 느슨한 이미지가 `ARENA`에 남아 있으면 원본 `ACD.EXE`와 한글 `ACDKR.EXE`가 모두 그 이미지를 읽어 화면이 섞인다. 별도 런타임은 파일명 참조를 빠뜨리는 위험 없이 두 실행 환경을 격리한다. `ARENA_KR`은 패치 배포물에 포함되는 게임 사본이 아니라 설치 대상 PC에서 정품 설치본으로부터 생성되는 로컬 결과물이다.

## 현재 알려진 제약

- 한글 입력기는 없으므로 캐릭터 이름 입력은 영문 중심이다.
- 한글은 현재 화면별 가변폭이 아니라 크기별 고정폭이다.
- 작은 영문 글꼴 화면은 최소 9픽셀 한글을 기준으로 UI 영역과 행간을 별도로 조정해야 한다.
- 컷신 자막은 화면 버퍼와 팔레트 전환을 모두 회귀 테스트해야 한다.
