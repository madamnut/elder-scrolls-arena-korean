# ACD.EXE 글자 렌더러 역공학 기록

분석 대상은 Steam판 Arena CD 1.07의 `ACD.EXE`이며 SHA-256은 다음과 같다.

```text
40dfed48a66154feeda2dc33b9549570b4392874750101f264e7de25ece98a7d
```

압축된 실행 파일은 PKLITE `1.12/l/e`로 식별된다. Deark 1.7.2로 해제하면 크기 321,728바이트인 MZ 실행 파일이 생성되며 SHA-256은 다음과 같다.

```text
3d698ac22c1f7da49d87c78d80f89f3c3822ba3f62708b67f98fff3dac300a86
```

MZ 헤더 크기는 `0x3F50`바이트이고 메모리에 적재되는 이미지는 305,520바이트다. 아래의 모든 주소는 파일 오프셋이 아니라 압축 해제된 MZ 적재 이미지 기준 오프셋이다. 파일 오프셋을 얻으려면 `0x3F50`을 더한다.

## 세그먼트와 글꼴 적재

- 주 데이터 세그먼트: 상대 세그먼트 `0x3BA0`
- 글꼴 적재 순서: `0x21F46`
- 범용 파일 로더 썽크: `0000:7504`
- 데이터 오프셋 `3A7C`, `3A87`, `3A92`, `3A9F`의 글꼴 파일명: `FONT_A.DAT`, `FONT_B.DAT`, `ARENAFNT.DAT`, `TEENYFNT.DAT`
- 적재된 글꼴 세그먼트 보관 위치: 데이터 오프셋 `1026`, `1028`, `102C`, `102E`
- 현재 글꼴 원거리 포인터: 데이터 `B034`(세그먼트 워드는 `B036`)
- 현재 글꼴 선택 함수: `0000:9AE6`
- 문자열 내부 글꼴 전환 제어: `0x0C` 다음에 1부터 시작하는 글꼴 번호
- 문자열 내부 색상 제어: `0x09` 다음에 색상 값

## 폭 계산

함수는 `0000:66F6`이며, 바로 앞에는 원거리 썽크가 있다.

- 입력: NUL로 끝나는 `DS:SI` 문자열
- 출력: `AX` 픽셀 폭
- 글꼴 높이: `font[0]`
- 공백 폭: `font[1] + 1`
- 그 밖의 출력 가능한 ASCII 폭: `font[character - 32]`
- 최상위 비트가 설정된 바이트는 `JS 6732`에서 명시적으로 건너뛴다. 이 지점이 첫 번째 한글 연결 지점이다.

## 비트맵 그리기

함수는 `0000:9E0A`이며, 바로 앞에는 원거리 썽크가 있다.

- 입력: `AX=x`, `BX=y`, NUL로 끝나는 `DS:SI` 문자열
- 프레임버퍼 세그먼트: 데이터 `A900`, 한 행은 320바이트
- 팔레트 인덱스: 데이터 `9288`
- 글리프 비트맵 시작 위치: `font + 0x5F + ((character - 33) * height * 2)`
- 각 행은 리틀 엔디언 16비트 워드이며 최상위 비트가 왼쪽 픽셀이다.
- 함수는 문자 값에서 32를 뺀 뒤 공백보다 작은 값을 거부하므로, 최상위 비트가 설정된 바이트도 렌더러에 도달할 수 없다.

행 배치 코드는 `0x9900`에서 폭 계산 함수를, `0x9967`과 `0x99C1`에서 그리기 함수를 호출한다. `0x99C5` 부근에서는 다음 행으로 이동할 값을 `font[0] + 1`로 읽는다.

## 연결 방식

현재 방식은 원본 실행 파일이 재배치한 데이터 세그먼트를 그대로 유지하고, 두 함수의 본체를 작은 `INT 60h` 스텁으로 교체한다.

- `DX=0`: 문자열 폭 계산. `DS:SI`는 문자열이고 `ES`는 Arena의 데이터 세그먼트다.
- `DX=1`: 문자열 그리기. `AX=x`, `BX=y`, `DS:SI`는 문자열이고 `ES`는 Arena의 데이터 세그먼트다.

DOS TSR이 인터럽트 `60h`를 담당한다. ASCII 동작은 원본과 같게 재현하고 프로젝트의 AKC 2바이트 한글 인코딩을 인식한다. Galmuri9 글리프는 32바이트 레코드(16개의 16비트 행)로 저장되므로 16KiB EMS 페이지 하나에 정확히 512개가 들어간다. TSR은 글리프 페이지를 매핑하기 전에 EMS 함수 `47h`로 전체 페이지 맵을 저장하고, 출력 후 함수 `48h`로 복원한다. 따라서 게임이 사용하던 EMS 매핑을 변경된 상태로 남기지 않는다.

## 현재 크기 조정 동작

`HANGUL.FNT`는 9×9 원본 비트맵을 저장한다. 현재 TSR은 큰 글꼴에서도 표본 추출에 사용하는 원본 높이는 9로 유지하면서, 화면에 표시하는 한글 높이를 최대 8행으로 제한한다. 높이가 8보다 작은 글꼴은 현재 Arena 글꼴 높이를 그대로 사용한다. 따라서 높이 6인 글꼴을 사용하는 직업 질문 화면은 큰 글꼴 제한의 영향을 받지 않으며, 행간을 더 확보하려면 작은 글꼴 전용 압축 정책이 따로 필요하다.

Arena의 영어 글리프는 가변폭이다. 일반 문자의 전진 폭은 활성 글꼴의 폭 표에서 읽지만, 현재 한글 경로는 활성 글꼴 크기마다 하나의 고정 전진 폭을 사용한다. 두 방식의 차이 때문에 같은 행 이동값을 사용해도 한글이 영어보다 조밀하게 보일 수 있다.
