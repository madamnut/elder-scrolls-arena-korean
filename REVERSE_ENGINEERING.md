# ACD.EXE 글자 렌더러 역공학 기록

분석 대상은 Steam판 Arena CD 1.07의 `ACD.EXE`이며 SHA-256은 다음과 같다.

```text
40dfed48a66154feeda2dc33b9549570b4392874750101f264e7de25ece98a7d
```

압축된 실행 파일은 PKLITE `1.12/l/e`로 식별된다. Deark 1.7.2로 해제하면 크기 321,728바이트인 MZ 실행 파일이 생성되며 SHA-256은 다음과 같다.

```text
3d698ac22c1f7da49d87c78d80f89f3c3822ba3f62708b67f98fff3dac300a86
```

MZ 헤더 크기는 `0x3F50`바이트이고 메모리에 적재되는 이미지는 305,520바이트다. 아래의 모든 주소는 파일 오프셋이 아니라 압축 해제된 MZ 적재 이미지 기준 오프셋이다. 파일 오프셋을 얻으려면 `0x3F50`을 더한다.

## 세그먼트와 글꼴 적재

- 주 데이터 세그먼트: 상대 세그먼트 `0x3BA0`
- 글꼴 적재 순서: `0x21F46`
- 범용 파일 로더 썽크: `0000:7504`
- 데이터 오프셋 `3A7C`, `3A87`, `3A92`, `3A9F`의 글꼴 파일명: `FONT_A.DAT`, `FONT_B.DAT`, `ARENAFNT.DAT`, `TEENYFNT.DAT`
- 적재된 글꼴 세그먼트 보관 위치: 데이터 오프셋 `1026`, `1028`, `102C`, `102E`
- 현재 글꼴 원거리 포인터: 데이터 `B034`(세그먼트 워드는 `B036`)
- 현재 글꼴 선택 함수: `0000:9AE6`
- 문자열 내부 글꼴 전환 제어: `0x0C` 다음에 1부터 시작하는 글꼴 번호
- 문자열 내부 색상 제어: `0x09` 다음에 색상 값

## 폭 계산

함수는 `0000:66F6`이며, 바로 앞에는 원거리 썽크가 있다.

- 입력: NUL로 끝나는 `DS:SI` 문자열
- 출력: `AX` 픽셀 폭
- 글꼴 높이: `font[0]`
- 공백 폭: `font[1] + 1`
- 그 밖의 출력 가능한 ASCII 폭: `font[character - 32]`
- 최상위 비트가 설정된 바이트는 `JS 6732`에서 명시적으로 건너뛴다. 이 지점이 첫 번째 한글 연결 지점이다.

## 비트맵 그리기

함수는 `0000:9E0A`이며, 바로 앞에는 원거리 썽크가 있다.

- 입력: `AX=x`, `BX=y`, NUL로 끝나는 `DS:SI` 문자열
- 프레임버퍼 세그먼트: 데이터 `A900`, 한 행은 320바이트
- 팔레트 인덱스: 데이터 `9288`
- 글리프 비트맵 시작 위치: `font + 0x5F + ((character - 33) * height * 2)`
- 각 행은 리틀 엔디언 16비트 워드이며 최상위 비트가 왼쪽 픽셀이다.
- 함수는 문자 값에서 32를 뺀 뒤 공백보다 작은 값을 거부하므로, 최상위 비트가 설정된 바이트도 렌더러에 도달할 수 없다.

행 배치 코드는 `0x9900`에서 폭 계산 함수를, `0x9967`과 `0x99C1`에서 그리기 함수를 호출한다. `0x99C5` 부근에서는 다음 행으로 이동할 값을 `font[0] + 1`로 읽는다.

## 연결 방식

현재 방식은 원본 실행 파일이 재배치한 데이터 세그먼트를 그대로 유지하고, 두 함수의 본체를 작은 `INT 60h` 스텁으로 교체한다.

- `DX=0`: 문자열 폭 계산. `DS:SI`는 문자열이고 `ES`는 Arena의 데이터 세그먼트다.
- `DX=1`: 문자열 그리기. `AX=x`, `BX=y`, `DS:SI`는 문자열이고 `ES`는 Arena의 데이터 세그먼트다.

DOS TSR이 인터럽트 `60h`를 담당한다. ASCII 동작은 원본과 같게 재현하고 프로젝트의 AKC 2바이트 한글 인코딩을 인식한다. Galmuri9 글리프는 32바이트 레코드(16개의 16비트 행)로 저장되므로 16KiB EMS 페이지 하나에 정확히 512개가 들어간다. TSR은 글리프 페이지를 매핑하기 전에 EMS 함수 `47h`로 전체 페이지 맵을 저장하고, 출력 후 함수 `48h`로 복원한다. 따라서 게임이 사용하던 EMS 매핑을 변경된 상태로 남기지 않는다.

## 현재 9픽셀 무축소 동작

`HANGUL.FNT`는 9×9 원본 비트맵을 저장한다. 2026년 7월 11일 1차 구현에서 활성 영문 글꼴 높이에 따라 5–8픽셀로 줄이던 최근접 축소 루틴을 제거했다. 한글은 항상 원본 9행과 가로 9픽셀을 그대로 그리고 오른쪽에 1픽셀을 비워 총 10픽셀 전진한다.

원본 행 배치 코드는 적재 이미지 오프셋 `0x99C4–0x99D5`에서 `font[0] + 1`을 계산했다. 이 18바이트를 `INT 60h`, `DX=2` 호출로 교체했다. TSR은 `DS:SI` 줄에서 유효한 AKC 쌍을 찾으면 `max(font[0] + 1, 11)`을 반환하고, ASCII만 있으면 원본 값을 반환한다. 따라서 `FONT_B`의 한글 줄은 11픽셀 간격을 사용하지만 영문 전용 줄은 기존 7픽셀 간격을 유지한다.

폭 계산과 그리기는 HANGUL9 한글 한 글자당 모두 10픽셀을 사용한다. 질문 파일 빌더도 같은 폭으로 줄바꿈하므로 계산값과 실제 커서 이동이 일치한다.

2026년 7월 12일 TSR을 3뱅크 구조로 확장했다. `font[0]`이 9 이하면 HANGUL9(폭 10·행 11), 10~12이면 HANGUL12(폭 12·행 14), 13 이상이면 HANGUL16(폭 17·행 18)을 선택한다. 세 뱅크는 각각 22 EMS 페이지이며 설치 때 총 66페이지를 적재한다. 질문 화면의 `FONT_B`는 높이 6이므로 기존 HANGUL9 동작을 그대로 유지한다.

## 추천 직업 결과 화면

질문 10개가 끝나면 실행 파일의 `Thou wouldst survive...` 형식 문자열에 추천 직업명을 `%s`로 삽입한다. 본문은 네 줄짜리 한국어 문구로 교체한다.

직업명은 실행 파일 안에 동일한 18개 NUL 종료 문자열 배열이 두 번 존재한다. 첫 배열은 인간형 적을 포함한 공용 이름표이고, 두 번째 배열은 캐릭터 생성용 직업 정의다. 원본 직업 체계와 외부 자료 대조를 쉽게 하기 위해 두 배열은 영어 원문을 유지한다.

추천 문구를 조립한 코드는 적재 이미지 `0x13F8E`에서 공용 Y/N 대화상자 래퍼 `0x16229`를 호출한다. 래퍼는 DS `0x4B60`의 입력 키 `YN`과 DS `0x4B63`의 포인터 표를 사용한다. 원본 포인터는 DS `0x4B69`의 `Yes\r`와 DS `0x4B6E`의 `No\r`를 가리키므로 선택지 글자는 IMG가 아니라 일반 글꼴 렌더러가 그린다.

한국어 패치는 초기 스택 꼭대기와 일치하는 원본 적재 이미지 끝에 AKC `예\r`, `아니요\r`를 추가하고 포인터 표만 새 문자열로 돌린다. 스택은 아래로 자라므로 추가 문자열과 충돌하지 않는다. 키 판정용 `YN`은 그대로 두어 조작은 원본과 같다. `YESNO.IMG`와 `ACCPREJT.IMG`는 이 결과창의 선택지 글자와 무관하다.

## 출신 지역 선택 지도

기본 지도와 영문 지역명은 느슨한 파일 `TAMRIEL.MNU`의 320×200 이미지에 함께 들어 있다. `GLOBAL.BSA`의 `OUTPROV.CIF`에는 지역별 금색 지역명 9개가 저장되며, 각 프레임 헤더의 X·Y는 프레임 캔버스 좌상단을 놓을 화면 픽셀 좌표다.

`OUTPROV.CIF`는 캐릭터 생성의 출신 지역 선택 화면에서 마우스 오버 효과로 사용되지 않는다. 해당 화면은 `TAMRIEL.MNU` 뒤쪽의 클릭 마스크로 클릭한 지역을 판정한 뒤 확인창을 띄운다. 금색 `OUTPROV.CIF` 프레임은 캐릭터 생성 이후 월드맵에서 현재 지역 하나를 표시할 때 사용된다. 따라서 출신 지역 선택 화면에서 커서를 움직여도 지역명이 변하지 않는 것이 원본 동작이다.

`TAMRIEL.MNU`는 이미지·팔레트 뒤 `0x87D5`부터 6,556바이트의 지역 클릭 마스크를 가진다. 원본 엔진은 마스크 시작점을 파일 헤더가 아니라 고정 오프셋으로 사용하므로, 지도 이미지를 단순 무압축 변환해 파일 길이를 늘리면 클릭 판정이 깨진다. 패치는 새 지도를 type-4로 다시 압축해 원본 33,993바이트 공간 안에 넣고 남는 영역을 채운 뒤, 내장 팔레트와 클릭 마스크를 원본 오프셋에 바이트 그대로 보존한다.

한글 지역명은 HANGUL9 9×9 글리프와 1픽셀 외곽선으로 만든다. 기본색은 원본 팔레트 119번, 강조색은 20번, 외곽선은 206번을 사용한다. `OUTPROV.CIF`는 한글 프레임의 새 크기와 사용자가 확정한 좌상단 좌표를 헤더에 기록한다.

지역 확정 뒤 능력치 배분 화면까지는 실행 파일의 공통 안내문 4개와 `TEMPLATE.DAT`의 종족별 레코드 8개가 조합된다. 공통 안내문의 언팩 ACD 이미지 오프셋은 `ConfirmedRace1=0x35E4C`, `ConfirmedRace2=0x3F8AA`, `ConfirmedRace3=0x35EA2`, `ConfirmedRace4=0x35EE0`이다. 두 번째 안내문에는 `TEMPLATE.DAT` 키 1409~1416 중 선택한 종족의 설명이 붙고, 세 번째 안내문에는 이미지 오프셋 `0x36034`부터 이어지는 직업별 선호 능력치 문구 18개 중 하나가 삽입된다. 패치는 공통 안내문, 종족 설명, 선호 능력치 문구를 모두 AKC로 바꾼다. 직업명은 합의한 정책에 따라 영어 원문을 유지한다.

두 번째 혈통 안내문만 앞뒤 공통 안내문과 출력 경로가 다르다. 원본은 `TEMPLATE.DAT` 레코드를 임시 버퍼로 읽고 493바이트를 이동한 뒤 제목을 앞에 붙이며, 40바이트 단위 자동 줄바꿈까지 다시 수행한다. AKC 2바이트 문자열은 이 특수 조립 경로에서 정상적인 앞뒤 안내문과 다른 결과가 나왔다. 패치는 이미지 오프셋 `0x1431E~0x143A8`의 조립 루틴을 교체하고, 8개의 완성된 `그대의 혈통:`+설명 문자열을 실행 파일 데이터 세그먼트 끝에 추가한다. 선택 종족 인덱스로 포인터를 고른 뒤 앞뒤 안내문과 동일한 공용 출력기 `0x730C`에 직접 전달한다.

후행 빈 줄로 상자를 키우면 그 빈 줄 자체가 아래 여백이 되므로 사용하지 않는다. 혈통 안내 루틴이 데이터 세그먼트의 전용 플래그를 켠 동안에만 상자 측정 지점 `0x99F2`가 줄 수에 HANGUL9 행 이동값 11을 직접 곱한다. 폭 후크 뒤 NOP 영역 `0x670D`의 도우미는 플래그가 꺼져 있으면 원본 `font[0]+1` 계산을 그대로 수행한다. 따라서 혈통 본문은 `n×11`로 측정·출력되고, 컷신과 다른 공용 텍스트는 원본 계산을 유지한다.

## 리아 실메인 첫 꿈 장면

새 게임 시작 루틴의 적재 이미지 `0x113E`는 `SI=0x809A`의 `vision.xmi`, `DI=0x8085`의 `vision.flc`, `AX=1400`을 설정한 뒤 컷신 실행기 `1194:D459`를 호출한다. 실행기는 `TEMPLATE.DAT` 키 1400을 읽고 `D540`에서 CR·LF를 NUL로 바꾸어 독립된 자막 행으로 나눈다.

FLC 재생 콜백 `2294:076C`는 남은 행 중 최대 3개를 가운데 정렬한다. 원본 좌표는 `(x=160,y=170)`, `(160,180)`, `(160,190)`이고 행 간격은 10픽셀이다. 다음 자막 시점에는 NUL 문자열 세 개를 건너뛰고 남은 행 수를 3만큼 줄인다. 원문은 33행이며 11개 구간이 모두 3행이다. 한국어도 빈 행 없는 33행을 정확히 유지해야 한다. 기존 34행 번역은 원본에 없는 12번째 구간을 만들었으므로 33행으로 바로잡았지만, 격리판 실기에서도 첫 조각 전 사망이 키 1403의 `.Perhaps I will even...`으로 이어졌다. 따라서 초과 구간은 오분기의 원인이 아니다. 한국어 각 행은 AKC 기준 최대 55바이트로 대응 원문 최대 63바이트보다 짧고, 전체도 1,437바이트로 원문 1,820바이트보다 작아 단순 고정 행 버퍼 초과도 배제한다.

점과 ASCII 이름만 보이던 첫 번째 원인은 컷신 템플릿의 마지막 문자 필터였다. 적재 이미지 `0x1C00B`의 unsigned `JBE`(`76 04`)는 소문자 `a`~`z`만 통과시키고 모든 AKC 바이트 `0x80`~`0xFF`를 버렸다. 이를 signed `JLE`(`7E 04`)로 바꾸면 소문자는 그대로 통과하고 `0x7B`~`0x7F`는 계속 거부하면서, 음수로 해석되는 AKC 선행·후행 바이트가 최종 자막 버퍼에 보존된다. 그 뒤 FLC 콜백 중 EMS 글꼴 호출이 안정적으로 끝나지 않는 별도 문제는 컷신 번역에 쓰이는 HANGUL9 음절을 TSR 상주 캐시에 두어 해결했다. 일반 UI는 캐시에 없는 글자를 기존 EMS 이동 경로로 계속 그린다.

원본 영어 `TEMPL_KR.DAT`를 같은 `ACDKR.EXE`·TSR과 조합한 A/B 판에서는 오프닝과 키 1402 사망 장면이 모두 정상이다. 처음에는 AKC 렌더러나 진행값 오염 가능성을 남겨 두었지만, 실패 화면의 영상은 계속 리아 실메인의 `VISION.FLC`였다. 사망 분기 `0x0D01`은 `P=0`일 때 `VISION.FLC`와 키 1402를 함께 선택하고, `P>0`이면 `JAGAR.FLC`와 키 1403을 함께 선택한다. 따라서 리아 영상과 제이거 탄 문장의 조합은 진행 분기가 아니라 텍스트 본문 주소 문제다. 오프닝 뒤 `P=0`을 다시 쓰는 진단 훅도 출력에 아무 영향을 주지 않아 제거했다.

원본 키 1400 헤더는 파일 오프셋 `0x4911D`, 키 1402 헤더는 `0x49999`, 본문 첫 바이트는 `0x499A0`, 키 1403 헤더는 `0x49A71`이다. 가변 길이로 만들었던 한글 파일은 키 1400이 원본 레코드보다 383바이트 짧았고 키 1402 헤더가 `0x4981A`로 당겨졌다. 그런데 런타임은 키 1402를 요청할 때 원본 본문 주소 `0x499A0`을 사용했다. 그 주소의 실제 한글 파일 바이트를 직접 읽으면 키 1403 네 번째 행의 `. Perhaps I\r\nwill even...`부터 정확히 이어진다. 사용자가 본 문구와 바이트 위치가 일치하므로 고정 본문 오프셋 파괴가 원인이다.

`apply_catalog.py`는 이제 `TEMPLATE.DAT`를 포함한 해시 레코드 번역을 가변 길이로 재배치하지 않는다. 번역이 원본보다 짧으면 표시 종료표식 `&` 바로 뒤에 ASCII 공백을 넣어 레코드의 원래 바이트 크기를 채운다. 번역이 원본 영역보다 크면 빌드를 중단한다. 출력 뒤에는 전체 파일 크기와 모든 `#키` 헤더의 `(키, 파일 오프셋)` 목록을 원본과 비교한다. 현재 키 1400·1402 격리판은 원본과 같은 395,981바이트이며 811개 헤더가 전부 같은 위치에 있다. 키 1402의 한글 본문도 원본 주소 `0x499A0`에서 시작하고 키 1403은 `0x49A71`에 그대로 남는다.

원본 `VISION.FLC`의 검은 자막 밴드는 Y=167~199의 33행이다. HANGUL9는 실제 9행을 모두 채우므로 원본 Y=170·180·190 배치에서는 행 사이와 아래쪽이 각각 1픽셀뿐이다. 한국어판은 영상의 검은 영역을 Y=163~199의 37행으로 확장하고, 출력 기준점을 Y=166·177·188, 행 간격을 11픽셀로 바꾼다. 글리프는 Y=166~174, 177~185, 188~196을 차지해 위·아래 3픽셀, 행 사이 2픽셀을 남긴다. 다음 프레임 전 지우기 사각형도 `(x=0,y=163,width=320,height=37)`로 맞춰 잔상을 막는다.

## 첫 본편 리아 실메인 꿈

첫 본편 꿈 진입 루틴의 적재 이미지 `0x1ED11`은 `SI=0x809A`의 `VISION.XMI`, `DI=0x4E16`의 `CHAOSVSN.FLC`, `AX=1500`을 설정해 혼돈의 지팡이 설명을 재생한다. 이어지는 `0x1ED3E`은 `DI=0x8085`의 `VISION.FLC`, `AX=1294`를 설정해 팽 레어 안내로 전환한다. 이후 지팡이 조각 진행도는 데이터 오프셋 `0x0F77`을 기준으로 키 1295~1302를 선택한다.

진행 장면은 영상 하나에 대사가 박힌 구조가 아니다. `0x1ED3E`은 `VISION.FLC`와 `AX=1294+[0x0F77]`을 결합하고, 제이거 탄 장면 `0x1F25C`는 `JAGAR.FLC`와 `AX=1392+[0x0F77]-1`을 결합한다. 사망 분기 `0x0D01`은 아직 지팡이 조각이 없으면 `VISION.FLC`·키 1402, 하나 이상 있으면 `JAGAR.FLC`·키 1403을 선택한다. 따라서 진행 대사 키 1295~1302·1392~1399와 두 사망 키를 모두 번역하고, 두 FLC 모두 동일한 Y=163~199 자막 밴드를 확보해야 한다.

## 공통 컷신 실행기와 장면별 캐시 훅

언팩 적재 이미지 `0x1ED9D`의 공통 FLC 실행기는 호출자가 선택한 템플릿 키를 `AX`로 받은 뒤 `0x1EDA5`에 저장하고, `0x1EDAB`의 재배치 far call로 원래 실행 루틴 `0x6E38`을 호출한다. 이 call의 오프셋만 검증된 코드 동굴 `0x6725`로 바꾼다. 10바이트 헬퍼는 `DX`를 보존한 채 `DX=3`, `INT 60h`을 호출하고 원래 `0x6E38`로 점프한다. 원래 far call의 반환 프레임은 그대로 남아 있고 원래 루틴의 `RETF`가 소비한다.

TSR은 66개 글꼴 EMS 페이지 뒤에 `CUTSCN.CCH`의 장면 페이지를 연속 배치한다. 키 표는 `(템플릿 키, 논리 EMS 페이지, 글리프 수)` 6바이트 레코드다. 훅 호출 시 선택 페이지의 `글리프 수 × 20`바이트를 고정 5,120바이트 일반 메모리 버퍼로 옮긴다. 각 항목은 16비트 한글 음절 인덱스와 HANGUL9의 9개 16비트 행이다. 항목을 인덱스 오름차순으로 저장하므로 영상 콜백은 EMS 호출 없이 이진 검색한다.

현재 검증에서는 26페이지의 실제 페이로드를 `HANGUL.FNT`와 항목별로 비교하고, 각 페이지의 미사용 꼬리가 전부 0인지 확인한다. `ACDKR.EXE`는 직전 패치본과 비교해 헬퍼 10바이트와 call 대상 2바이트만 다르다.

키 1500은 원문과 같은 15행·5화면, 키 1294는 16행·6화면으로 조판한다. 플레이어 성별형 이름 `%pcf`와 이름 `%pcn` 치환자는 각각 한 번씩 보존한다. `CHAOSVSN.FLC`는 320×200, 31프레임, 프레임당 114ms이며 원본 그림은 Y=0~166까지 존재한다. 한국어판은 자막과 겹치지 않는 Y=163부터 검게 덮으므로 최종 비교에서는 Y=0~162의 픽셀과 모든 프레임 팔레트·재생 속도가 원본과 정확히 같아야 한다.

## 첫 던전과 공통 인게임 문구

시작 던전의 환경 설명 여섯 개는 실행 파일이 아니라 `GLOBAL.BSA/START.INF`의 `@TEXT` 블록에 있다. `apply_catalog.py`가 UTF-8 번역을 AKC로 바꾼 평문 INF를 만든 뒤, 기존 한글 `GLOBAL_K.BSA`를 기준으로 `START.INF`만 다시 암호화·교체한다. 원본 BSA부터 새로 만들면 이미 적용한 지도와 캐릭터 생성 이미지가 사라지므로 금지한다.

문·자물쇠·열쇠·상자·금화·인벤토리·장비·휴식·야영·저장·이동·기초 주문·장비 상세 안내는 언팩 ACD 적재 이미지의 NUL 종료 문자열이다. 현재 130개를 `INGAME_MESSAGE_PATCHES`로 관리한다. 각 항목은 원본 오프셋과 원문 전체를 검증하고, `%s`, `%u`, `%d` 같은 C `printf` 자리표시자의 종류와 개수를 비교하며, AKC 결과가 원문 바이트 예산을 넘으면 빌드를 중단한다. 남는 공간은 NUL 뒤를 0으로 채워 이후 문자열의 주소를 움직이지 않는다.

휴식 메뉴의 제목·시간 경과·회복·기상·적 접근 문구는 일반 NUL 문자열이지만 두 선택지는 다르다. 적재 이미지 `0x4034E`의 `Camp for a while...`과 `0x40367`의 `Until fully healed` 앞에는 색상 제어 바이트와 단축키 `C/U`가 끼어 있다. `CONTROLLED_INGAME_MESSAGE_PATCHES`는 제어 바이트와 입력 키를 보존한 채 표시 부분만 `시간 지정 휴식`, `완전 회복까지`로 바꾼다.

마을 주민 대화의 도시별 인사말은 실행 파일에 있지 않고 느슨한 해시 레코드 파일 `CITYTXT`에 있다. 키 `0600`~`0663`은 64개 도시와 일대일로 대응한다. `city-greetings-overrides.json`은 원문과 같은 행 수를 유지하고, `apply_catalog.py`는 각 `&` 뒤를 패딩해 원본 10,669바이트와 64개 헤더 위치를 보존한다. 현재 한국어판의 가장 긴 행은 HANGUL9 실제 전진폭 184픽셀이다.

주민에게 말을 걸면 먼저 적재 이미지 `0x43F23`의 `ASK ABOUT?` 메뉴가 열린다. `Who are you?`, `Where is...`, `Rumors`, `Exit`와 소문 종류 `General/Work`는 색상 제어 바이트 사이에 단축키 `W/H/R/E`, `G/W`가 들어간 문자열이다. 표시문만 번역하고 키는 그대로 둔다. `I have nothing further to discuss!`, 도시 안·밖에서 다시 물으라는 답과 소문 거절문은 일반 고정 문자열로 번역한다. 여관 주인의 방값 흥정은 `TAVERN.DAT` 75개 레코드를 사용하는 별도 계통이므로 이 묶음에 포함하지 않는다.

메뉴를 선택한 뒤 나오는 실제 주민 답변은 `CITYTXT`나 실행 파일 고정 문자열이 아니라 `TEMPLATE.DAT`에 있다. 키 `0100`~`0114`는 NPC 직업별 첫 `누구인가?` 답변, `0115`~`0129`는 같은 NPC에게 다시 물었을 때의 답변이다. 키 `0159`는 일반 소문 없음, `0160`은 일거리 없음, `0162`~`0168`은 도시 사건 소문, `0169`~`0180`은 축일 소문 문장 조각이다. 키 `0161`은 원본 카탈로그에도 없으며 번호가 바로 `0162`로 이어진다. 각 레코드에는 `&`로 끝나는 무작위 변형이 5~10개 있으므로 일부 문장만 바꾸면 영어가 간헐적으로 다시 나온다. `npc-dialogue-overrides.json`은 이 51개 레코드의 모든 변형을 번역한다. 이름·종족·도시 같은 `%n`, `%ra`, `%cn` 자리표시자는 출현 횟수를 보존한다. 반면 `%oc`, `%doc`, `%jok`, `%oth`는 별도의 대규모 영문 절차 생성표를 삽입하므로, 이미 같은 의미를 완결한 한국어 문장에서는 의도적으로 생략해 영문 조각이 섞이지 않게 한다.

`장소 묻기`의 목록과 답변도 다시 둘로 나뉜다. 도시 안 목록 9개는 적재 이미지 `0x43FD3`, 도시 밖 목록 9개는 `0x44035`의 NUL 연속 문자열 표다. `여관`, `사원`, `장비점`, `마법사 길드`, `궁전`, `성문/일거리`, 가까운 시설 세 항목으로 원래 개수와 순서를 유지해 다시 묶는다. 실제 답변은 `TEMPLATE.DAT` 키 `0259`의 위치 모름 8종, `0260`의 방향 안내 8종, `0261`의 지도 표시 8종이다. `%di`가 삽입하는 8방향 표는 적재 이미지 `0x39CC6`에 있으며 북쪽부터 북서쪽까지 원본 순서로 번역한다.

훔치기 버튼의 `Select target for pilfering...`은 적재 이미지 `0x4155C`의 고정 문자열이다. 상태 버튼의 아이콘 자원명은 `STATUS.CIF`지만, 본문은 이미지에 박혀 있지 않다. 적재 이미지 `0x4157B`의 형식 문자열이 현재 장소·시각·날짜·현재 중량·최대 중량을 `%s`, `%d`로 받아 실행 중에 조합한다. 원문에는 세 번째 `%s` 바로 뒤에 다음 문장의 `You`가 붙어 있으므로 자리표시자 검증기는 뒤따르는 일반 문자를 포함하지 않고 실제 변환 지정자 하나만 읽어야 한다. 상태창 안에 끼워 넣는 시각 표현 7개, 달 이름 12개, 요일 7개, 날짜 형식과 영문 서수 접미사도 각 원래 포인터 위치에서 함께 번역한다.

아이템 이름은 한 문자열이 아니라 재질·기본 이름·마법 접미사를 조합해 만든다. 예를 들어 언팩 적재 이미지 `0x3E03F`의 재질 표, `0x3DC04`의 무기 표, `0x3DD1F`의 무기 마법 표를 차례로 선택하면 `Iron Dagger of Strength`가 된다. 공용 루틴 `1194:559F`는 `SI=표 시작`, `CX=인덱스`를 받아 NUL 문자열을 순서대로 건너뛴다. 따라서 `GAMEPLAY_NAME_TABLES`는 항목 수와 순서를 고정한 채 표 전체를 원래 총 바이트 범위 안에서 다시 묶는다. 무기 마법은 영어식 후치 `of Strength` 대신 조합 뒤에도 자연스러운 `(힘)`을 사용한다.

몬스터 표 `0x36DBE`에는 몬스터 23종 뒤에 직업 19종이 연속 저장되어 있다. 같은 고정 개수 표에서 몬스터만 번역하고 직업 구간은 영어를 다시 기록한다. 열쇠는 별도 표 `0x3EE2B`에 `a Brass`, `an Iron`, `a Ruby`처럼 관사까지 포함되어 있으며, 호출부 `0x24DE0`이 12개 중 하나를 고른다. 이 표를 `황동`, `철`, `루비`처럼 바꾸면 기존 `%s key` 문구가 `루비 열쇠`로 완성된다. 장비·물약·내구 상태까지 총 135개 표시명을 번역하며, 빌드 때 원문 표 전체와 AKC 총 길이를 검증한다.

바닥·상자·시체의 공용 아이템 목록은 `(x=11,y=145,width=298,height=31)`의 고정 영역을 사용한다. 원본은 Y=145·154·163의 9픽셀 간격으로 최대 3개를 그리고 `...More`를 Y=169에 겹쳐 표시할 수 있다. HANGUL9는 9행을 모두 사용하므로 한국어판은 페이지당 항목을 2개로 제한하고 Y=145·156의 11픽셀 간격으로 그리며, 다음 페이지가 있으면 Y=167에 `...More`를 표시한다. 수량 입력 커서도 같은 11픽셀 배수를 사용한다. 이렇게 하면 세 영역이 모두 Y=145~175의 기존 31픽셀 뷰포트 안에 들어간다.

자동지도는 고정 이미지와 실행 중 출력이 겹치는 혼합형이다. `AUTOMAP.IMG`에는 양피지 배경, `Exit`, 나침반 `N/E/S/W`가 픽셀로 박혀 있다. 반면 상단의 현재 장소명, 탐색한 지도 선, 플레이어 방향 표시는 실행 파일이 그 위에 동적으로 그린다. 시작 던전의 장소명은 적재 이미지 `0x405DB`의 `Imperial Dungeons`이며 표시 문자열만 `제국 지하감옥`으로 바꾼다. 따라서 이후 도시·던전명까지 한글화하려면 `AUTOMAP.IMG`를 다시 만들 것이 아니라 활성 장소명이 공급되는 데이터와 런타임 이름 표를 번역해야 한다.

일지의 `Journal`, `print`, `Exit`는 `LOGBOOK.IMG`에 박힌 픽셀이다. 장비 화면은 실행 파일이 실제로 참조하는 171×200 `EQUIP.IMG`에 `Level`, `EQUIPMENT`, `EXIT`, `SPELLBOOK`, `DROP`이 박혀 있고, 아이템명·상태·수치는 실행 파일이 그 위에 동적으로 출력한다. `EQUIP.IMG`에는 자체 팔레트가 없으므로 원본 `CHARSHT.COL`의 8바이트 헤더 뒤 RGB 768바이트를 그대로 사용한다. `localize_ingame_ui.py`는 세 이미지의 크기와 좌표를 보존하고 HANGUL9·HANGUL16 원본 비트맵으로 고정 글자를 다시 그린다. 버튼 클릭 판정은 별도이므로 이미지 좌표를 바꾸지 않는다.

## 능력치 배분 화면

능력치 화면은 이미지 한 장으로 완성된 UI가 아니라 고정 이미지와 실행 파일 출력이 겹치는 혼합형이다.

- `GLOBAL.BSA`의 `CHARSTAT.IMG`(171×200): 능력치·상태 고정 라벨과 `DONE`
- `GLOBAL.BSA`의 `BONUS.IMG`(77×16): `BONUS POINTS` 고정 라벨
- 실행 파일: 이름·종족·직업, 모든 수치, 선택 화살표와 팝업 문장
- `UPDOWN.IMG`: 증감 조작 그림이며 번역할 문자는 없다.

세 자원은 `CHARSHT.COL`을 공유한다. 이 776바이트 파일은 8바이트 헤더 뒤에 이미 0~255 범위인 RGB 768바이트를 저장한다. 일반 IMG 내장 VGA 팔레트처럼 0~63에서 다시 확대하면 63보다 큰 성분이 모두 흰색 쪽으로 포화되어 청록색이 지나치게 밝아진다. 스탯창 PNG 미리보기는 헤더만 건너뛴 원시 RGB 값을 그대로 사용해야 한다.

고정 라벨은 HANGUL9를 축소하지 않고 다시 그린다. 최종 작업물은 기본 능력치를 `힘·지능·의지·민첩·속도·인내·매력·행운`으로 표기하며 첫 행 Y=36, 행 간격 11픽셀을 사용한다. 기본 능력치 수치, 선택 화살표, 클릭 레코드도 원본 8픽셀 격자에서 같은 11픽셀 격자로 함께 옮긴다. 라벨 이미지만 바꾸면 화살표와 클릭 대상이 어긋날 수 있으므로 실행 파일의 값 출력 좌표·화살표 트랙·부분 다시 그리기 영역도 같은 좌표계로 패치한다.

최종 `BONUS.IMG`는 77×16 크기를 유지하고 헤더 좌표를 원본 `(45,109)`에서 `(62,119)`로 옮긴다. 잔여 스탯 숫자는 이미지 내부 상대 위치 `(48,6)`, 즉 화면 절대좌표 `(110,125)`에 출력한다. 고정 이미지 두 장은 모두 `CHARSHT.COL`에 실제로 존재하는 RGB만 사용하며 임의 재양자화하지 않는다.

동적 숫자는 좌상단 기준의 공통 격자를 사용한다. 위쪽 세 열의 X는 `25, 86, 147`, 여덟 행의 Y는 `38 + 11×행 번호`다. 아래 상태 영역은 X `25, 86`, Y `139, 150, 161`을 사용한다. 빈 라벨 행도 11픽셀 슬롯을 차지하며 뒤 항목을 당겨 올리지 않는다.

| 행 | 좌측 `(x,y)` | 중간 `(x,y)` | 우측 `(x,y)` |
|---:|---|---|---|
| 1 | 힘 `(25,38)` | 피해 `(86,38)` | 소지 한도 `(147,38)` |
| 2 | 지능 `(25,49)` | 마력 `(86,49)` | 비움 |
| 3 | 의지 `(25,60)` | 마법 저항 `(86,60)` | 비움 |
| 4 | 민첩 `(25,71)` | 명중 `(86,71)` | 회피 `(147,71)` |
| 5 | 속도 `(25,82)` | 비움 | 비움 |
| 6 | 인내 `(25,93)` | 성장 체력 `(86,93)` | 회복률 `(147,93)` |
| 7 | 매력 `(25,104)` | 카리스마 `(86,104)` | 비움 |
| 8 | 행운 `(25,115)` | 비움 | 비움 |

아래 상태 영역은 체력 `(25,139)`, 활력 `(25,150)`, 금화 `(25,161)`, 경험치 `(86,139)`, 레벨 `(86,150)`이다. 숫자는 `CHARFNT.DAT`의 4×8 전진 셀을 사용하며 실제 `0` 획은 3×6픽셀이다.

실행 파일에는 능력치명 8개와 종족명 단수·복수 표가 NUL 종료 문자열 표로 들어 있다. 항목별 원래 공간에 억지로 맞추지 않고 표 전체의 끝은 유지한 채 각 문자열의 시작 포인터를 다시 계산한다. 저장·다시 굴림 선택지는 원본 단축키 제어 바이트를 유지하고 표시 문자열만 `S 저장`, `R 다시 굴림`으로 바꾼다.

능력치 배분부터 외형 선택까지 사용하는 안내문도 실행 파일 문자열이다. 저장할 때 보너스 점수가 남았다는 경고, 외형 변경 방법과 완료 안내를 AKC 문자열로 교체한다. 외형 그림 자체와 얼굴 좌우 전환 자원에는 번역할 문자가 없다.

## 메인 메뉴 클릭 영역

타이틀 뒤 메인 메뉴의 글자는 `GLOBAL.BSA`의 320×200 `MENU.IMG`에 박혀 있지만 클릭 판정은 실행 파일의 별도 메뉴 레코드 세 개가 담당한다. 레코드는 적재 이미지 `0x38A60`부터 0x62바이트 간격으로 놓이고, 첫 8바이트가 `x, y, width, height` 네 개의 16비트 값이다. 판정은 오른쪽·아래 끝점을 포함하므로 Y=51~65는 `y=51, height=14`로 저장한다.

| 항목 | 원본 Y 범위 | 한글 이미지 Y 범위 | 저장값 `(y, height)` |
|---|---:|---:|---:|
| 저장한 게임 불러오기 | 0~63 | 51~65 | `(51, 14)` |
| 새 게임 시작 | 110~130 | 104~118 | `(104, 14)` |
| 종료 | 143~173 | 151~164 | `(151, 13)` |

세 레코드의 X는 모두 0, 폭은 320이므로 화면 가로 전체가 클릭된다. 항목 번호는 불러오기 0, 새 게임 1, 종료 2이며 단축키 `L`, `S`, `E`는 그대로 유지한다.

## 직업 질문 화면의 조판 기준

질문 화면은 런타임 자동 줄바꿈을 하지 않는다. 원본 `QUESTION.TXT`의 515줄은 모두 CRLF가 아니라 `공백(0x20) + LF(0x0A)`로 끝나며, 엔진은 이 형식으로 구분된 줄을 폭 계산·출력 함수에 전달한다. 다음 줄의 Y 좌표는 바로 앞 줄의 행 이동값을 더해 결정하므로 질문과 세 선택지의 위치도 앞에 놓인 줄 수에 따라 함께 내려간다.

`FONT_B.DAT`는 높이 6픽셀, 문자별 전진폭 2–6픽셀, 공백 전진폭 3픽셀이다. 원본 질문 40개를 이 폭표로 다시 계산하면 가장 넓게 배치되는 줄은 뒤쪽 공백까지 299픽셀이고, 가장 긴 11번 문항은 18줄 × 7픽셀로 126픽셀을 사용한다. 한글 질문 빌더는 이 두 값을 원본 화면의 가로·세로 예산으로 사용한다.

질문 전용 직접 그리기는 X=20에서 시작한다. 원본 299픽셀 폭을 그대로 허용하면 오른쪽 끝이 X=319에 닿으므로 한국어 빌더는 표시 경계를 X=310으로 두고 후행 공백까지 최대 290픽셀로 줄바꿈한다. 따라서 오른쪽에 10픽셀 여백이 남는다. 이 설정에서 현재 번역은 최대 9줄, 99픽셀로 원본 세로 예산 126픽셀 안에 들어간다.

이 텍스트 루틴에서 CR과 LF는 동등하지 않다. CR(`0x0D`)은 종료한 줄에 가운데 정렬 플래그를 설정하지만 원본 질문 파일의 LF는 그 플래그를 설정하지 않는다. 또한 원본의 판정 태그 120개는 모두 정확히 `(5x) + 공백 + LF`로 끝난다. 초기 한국어 빌더는 CRLF를 넣은 뒤 LF 전용으로만 바꾸면서도 후행 공백을 제거해, 시작 문항은 찾지만 현재 문항의 `c)` 종결을 찾지 못하고 이후 문항을 EOF까지 같은 두루마리에 붙였다. 빌더는 모든 줄을 원본과 같은 `공백 + LF`로 생성하고 그 후행 공백까지 299픽셀 폭 예산에 포함한다.

질문 오버레이의 선택 루틴은 적재한 질문 파일에서 공백을 건너뛴 뒤 십진수 번호를 읽어 원하는 문항의 시작 포인터를 찾는다. 화면 준비 루틴은 그 포인터부터 먼저 ASCII 콜론 `:`을 검색하고, 콜론 뒤에서 세 번 `(`를 찾아 각 `(5l)/(5c)/(5v)` 태그를 공백으로 지운다. 세 번째 태그의 `(`에서 6바이트 뒤, 즉 `(5x) + 공백 + LF` 바로 다음에 있는 다음 문항 첫 숫자를 임시 NUL로 바꿔 현재 문항만 잘라낸다. 화면이 끝나면 저장한 숫자를 복원한다.

원문 40개는 모두 선택지 전에 콜론이 정확히 하나 있지만 초기 한국어 번역은 자연스러운 물음표·마침표로 바꾸면서 콜론을 전부 제거했다. 이 경우 엔진의 콜론 검색이 현재 문항을 넘어 파일 뒤쪽까지 진행해 여러 문항이 하나의 두루마리에 연결된다. 빌더는 질문 본문의 마지막 문장부호를 콜론으로 정규화하고, 인코딩 결과에서 `번호 → 콜론 → 태그 3개 → 다음 번호` 경계를 엔진과 같은 바이트 오프셋으로 40개 모두 검증한다.

혼합 문자열의 가로 전진폭은 문자별 활성 폰트를 따른다. 반면 세로 행 이동값은 줄마다 나타난 문자 종류로 바꾸지 않고 한글화된 텍스트 블록에 선택된 한글 뱅크를 기준으로 고정하는 것을 최종 규칙으로 삼는다. 현재 `DX=2` 구현은 각 줄에서 AKC를 탐지하는 전환 단계이므로, 이후 화면별 한글 뱅크 선택 정보와 연결해 블록 안의 ASCII 전용 줄에도 같은 행 이동값을 반환해야 한다.

현재 질문 빌더는 `(5l)`, `(5c)`, `(5v)` 판정 태그가 혼자 다음 줄로 넘어가면 앞줄의 마지막 한글 단어를 태그와 함께 이동시킨다. 빌드 결과에 한글이 없는 줄이 하나라도 남으면 실패하므로 질문 화면의 219줄은 모두 HANGUL9 기준 11픽셀 행 이동값을 사용한다.

그리기 인터럽트는 줄을 한 번 미리 훑어 AKC 한글 포함 여부를 기록한다. 혼합 줄에서 활성 영문 글꼴 높이가 9픽셀보다 작으면 영문 비트맵의 시작 주소를 `320 × (9 - font[0])`만큼 아래로 보정한다. `FONT_B`는 높이 6픽셀이므로 `a)`, 괄호, 숫자와 판정 태그가 한글보다 3픽셀 아래에서 그려져 두 글꼴의 셀 아래쪽이 일치한다. 커서의 가로 전진 위치에는 이 보정값을 남기지 않는다.

원본 직접 그리기 함수 `0000:9E0A`는 `LODSB`로 종료 NUL까지 소비하고 입력 `SI`를 보존하지 않는다. 따라서 반환 시 `SI`는 NUL 바로 다음 바이트를 가리킨다. 질문 스크롤 초기화 루틴은 LF를 여러 NUL 문자열로 바꾼 뒤 같은 `SI`로 그리기 함수를 반복 호출하면서 이 부작용을 이용해 다음 줄로 이동한다. 초기 TSR이 입력 `SI`를 복원하자 첫 빈 문자열만 반복 출력되어 화면이 비었고, 스크롤의 별도 한 줄 갱신 경로에서만 일부 문장이 나타났다. `DX=1` 처리기는 이제 저장한 입력 `SI`를 버리고 소비한 NUL 다음의 `text_offset`을 반환한다.

질문 스크롤 전용 루틴은 일반 문단 행 이동 훅과 별도로 8픽셀 행 슬롯과 48픽셀 표시 복사 영역을 사용했다. HANGUL9 전용 패치는 표시 영역 48픽셀은 유지하면서 행 슬롯을 11픽셀로 바꾸고 다음 값을 함께 조정한다.

| 항목 | 원본 | HANGUL9 패치 |
|---|---:|---:|
| 준비 슬롯 수 | 8 | 6 |
